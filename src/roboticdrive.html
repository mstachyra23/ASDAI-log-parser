Inter-process communication¶

This page lists every topic which is published by RoboticDrive, and how it is used.

Note: Every message will have a topic, sender_id, and timestamp.

Messages may also have a recipient_id. Messages with a recipient_id should be ignored if the UUID does match your own sender_id.

CONSOLE_STATUS_UPDATE¶

This message is used for application log messages, which are primarily used for debugging.

Messages have an associated level, which is one of the following strings, ordered from highest-to-lowest:

  • CRITICAL Only used when something has gone wrong and the program is unable to continue. May be shown in the VR interface. This is the highest level.
  • ERROR Something has gone wrong and the program is in a substantially different state than the operator would expect. May be shown in VR interface.
  • WARNING A desired action may not have been performed. May be shown in the VR interface.
  • NOTICE A message that the user may want to see about “correct” behavior. May be shown in the VR inteface. We may later aim to move all information from this level into dedicated UI elements.
  • INFO Highest log level used inside libraries.
  • DEBUG Useful information for programmers or debugging errors at their source.
  • TRACE Fine-grained debug messages. Disabled by default as they may be used excessively. This is the lowest level.
Keys:
  • text (string) – Message contents, without any of the following metadata.
  • level (string) – Indicates the message severity, see description above for levels.
  • source (string) – Name of the program sending this topic.
  • timestamp (double) – Unix timestamp (seconds) of when the message was generated.
  • function (string) – (optional) Name of function generating the message.
  • filename (string) – (optional) Name of file generating the message.
  • line (int) – (optional) Line number where the message was generated.
  • error (int) – (optional) Linux errno(3) related to the message.
  • pid (int) – (optional) Process id of the process generating the message.
Publishers:

Any process – This message can be sent by any process.
Subscribers:

RoboticDriveUI – Uses source and text to compose a message to display. Uses type to format that string.

      physical:button_3 :logical: grasper_lock :type: bool
      physical:button_4 :logical: camera_zoom :type: bool

      physical:dpad_up :logical: elbow_angle_up :type: bool
      physical:dpad_left :logical: wrist_negative :type: bool
      physical:dpad_right :logical: wrist_positive :type: bool
      physical:joystick_click :logical: camera_reset :type: bool
      physical:joystick_x :logical: camera_x :type: double
      physical:joystick_y :logical: camera_y :type: double

HUMAN_ARM_POSE¶
Keys:
  • position (double[3]) – Position (mm) of human hand relative to human origin.
  • orientation_quat (double[4]) – Orientation of human hand relative to human origin frame.
  • pointer_orientation (double[4]) – Orientation to be used for the virtual pointer (not coupled with grasper angle).
  • sensor_position (double[3]) – Position (mm) of hand controller relative to chest frame
  • sensor_orientation_quat (double[4]) – Orientation of hand controller relative to chest frame
  • distortion_lvl (string) – Indication of how noisy the data in position, orientation_quat, and grasper_angle are. “GOOD”, “OKAY”, and “BAD” are used if published by polhemus, while only “GOOD” and “BAD” are used if published by the VRApplication. Similarly, “GOOD” and “BAD” are published if using the linked controllers to indicate a successful pose read.
  • grasper_angle (string) – The normalized fraction between 0 and 1 corresponding to the angle of the finger actuated joint on the linked hand controllers. It is the proportion of how opened the grasper is on the input device. It is turned into an actual angle (in radians) by armdrive. Will only be part of this message if using linked hand controllers.
  • human_role (string) – Indicate what the arm should control on the robot. Ex. "arm_right"
  • elapsed_time (double) – Time (seconds) between this publication and the first publication.
  • sender_id (string) – UUID generated on startup by the application sending this topic.
  • using_linked_controller (string) – Boolean indicating whether or not the pose source is Force Dimension linked hand controllers (true) or Polhemus free hand controllers (false).
Publishers:
  • hand_controller – Publishes this topic whenever it receives a new packet of raw position and orientation data, and if using linked hand controllers, grasper angle.
  • VRApplication – Publishes both of these topics whenever it receives a new packet of raw position and orientation data.
Subscribers:
  • armdrive – Scales and transforms position, orientation_quat, and grasper_angle to the robot frame. Uses the results as conditions in inverse kinematics to determine a set of robot angles that results in robot end effectors matching the human hands.
  • RoboticDriveUI – Uses distortion_lvl to visualize the polhemus or VRApplication distortion.
HAND_CONTROLLER_MODES¶
This message contains:¶
- The perceived modes of the system as observed by hand_controller_sm.c¶
- The commands to give to the linked controllers as a result of the modes¶
- The sources connected to the hand controller application.¶
Keys:
  • system_modes (object) – The perceived modes of the system, detected by parsing IPC messages. Has keys: :key bool vr_menu_open: True if the hand controller application thinks the VR menu is open (observed from IPC) :key bool cleaning: True if the hand controller application thinks the camera is in the cleaning state (observed from IPC) :key bool camera_mode: True if the hand controller application detects the system is in camera mode (from IPC) :key bool scan_mode: True if the hand controller application detects the system is in scan mode (from IPC) :key bool grasper_aligning: True if the hand controller application detects the system is aligning chest to graspers (from IPC) :key bool left_clutching: True if the hand controller application detects the left arm is clutching while NOT in camera, scan, or grasper alignment modes :key bool right_clutching: True if the hand controller application detects the right arm is clutching while NOT in camera, scan, or grasper alignment modes :key bool left_engaging: True if the hand controller application detects the left arm is trying to engage :key bool right_engaging: True if the hand controller application detects the right arm is trying to engage
  • linked_commands (object) – The commands issued to the linked hand controllers by the hand_controller application. Has keys: :key bool lock_rotation_left: True if the left hand should be locked rotationally. :key bool lock_rotation_right: True if the right hand should be locked rotationally. :key bool lock_translation_left: True if the left hand should be locked translationally. :key bool lock_translation_right: True if the right hand should be locked translationally.
  • sources_connected (object) – Which sources are currently connected to the hand controller applications. Has keys: :key bool source_vr_connected: True if hand_controller is receiving messages from VR. :key bool source_rss_connected: True if hand_controller is receiving messages from rssdrive. :key bool source_camera_conncted: True if hand_controller is receiving messages from cameradrive. :key bool source_robot_left_connected: True if hand_controller is receiving messages from armdrive left. :key bool source_robot_right_connected: True if hand_controller is receiving messages from armdrive_right.
Publishers:

hand_controller – Publishes this topic periodically.

ROBOT_ARM_MATCHING_HUMAN¶
Keys:
  • position (double[3]) – Position of human hand relative to human origin – scaled down to robot scale &amp; transformed with any robot world drag transformations.
  • orientation_quat (double[4]) – Orientation of human hand relative to human origin frame – transformed with any robot world drag transformations.
  • distortion_lvl (string) – Indication of how noisy the data in position, orientation_quat, and grasper_angle are. “GOOD”, “OKAY”, and “BAD” are used if published by polhemus, while only “GOOD” and “BAD” are used if published by the VRApplication.
  • elapsed_time (double) – Time (seconds) between this publication and the first publication.
  • sender_id (string) – UUID generated on startup by the application sending the origin of this key.
  • configuration (double[8]) – Joint values of robot arm derived from human arm pose - not necessarily currently being used to drive the robot.
  • overreach (optional) (double) – How far beyond the robot’s max reach (in milimeters) the human is attempting to command the robot. Value is positive when user is reaching past the max reach and negative when they are within the max reach.
  • underreach (optional) (double) – How far into the camera protection paraboloid (in milimeters) the human is attempting to command the robot. Value is positive when user is reaching into the paraboloid and negative when they are outside of the protection.
  • backreach (optional) (double) – How far behind the robot shoulders (in milimeters) the human is attempting to command the robot. Value is positive when user is reaching behind the shoulders and negative when they are in front of the shoulders.
  • latency (double) – Time in sec between this update and the previous update.
  • elbow_angle (double) – The angle of the robot arm elbow associated with the published pose.
  • grasper_angle (double) – The angle of the robot arm grasper associated with the published pose.
Publishers:

armdrive – Publishes the corresponding left and right version of these topics whenever it receives a valid topic HUMAN_ARM_POSE
Subscribers:
  • RoboticDriveUI – Uses configuration to visualize the human arms (white) and update the angle table.
  • VRApplication – Uses configuration to visualize human arms (white). Uses position and orientation_quat during instances of clutch out for human hand indicators.

ROBOT_ARM_STATUS¶
Keys:
  • close_enough_position (bool) – Indicates whether the robot arms matching the human are close enough to the last commanded robot position, ignoring orientation.
  • close_enough_orientation (bool) – Indicates whether the robot arms matching the human are close enough to the last commanded robot orientation, ignoring position.
  • close_enough_grasper (bool) – Indicates whether the grasper angle matching the human is close enough to the last commanded robot grasper angle.
  • timed_out (bool) – Indicates whether the last received human data was too long ago.
  • tracking_error_position (optional) (double[3]) – A 3-vector representing how far (in mm) the matching human end effector is from the calculated end effector position based on the sensed angles of the robot arm.
  • tracking_error_position_intensity (optional) (double) – A normalized value indicating how bad (intense) the tracking error is. A value of 1 indicates that there is a substantial difference between the commanded and sensed end effector positions. A value of 0 indicates the difference is minimal.
  • tracking_error_orientation (optional) (double[4]) – A 3-vector representing how far off (in rad) the matching human end effector is from the calculated end effector orientation based on the sensed angles of the robot arm.
  • tracking_error_orientation_intensity (optional) (double[3]) – A 3-vector of normalized values indicating how bad (intense) the tracking error is. A value of 1 indicates that there is a substantial difference between the commanded and sensed end effector orientations. A value of 0 indicates the difference is minimal.
  • intent_to_clutch_in_progress (double) – Value between 0.0 and 1.0 indicating how long the human configuration has been close enough to the robot configuration. Set to 1 when driving, 0 when intent_to_clutch_in is false.
  • latency (double) – Time (seconds) between timestamp of the message received (arm) and when that message is processed.
  • motion_type (string) – The current mode of motion the robot arm is currently in. Will be “instrument”, “clutch”, or “global_frame_control”.

This command is only valid in the state DRIVING. To enter DRIVING, send a ROBOT_SUPPORT_COMMAND with command DRIVE. Insertion stops when a STOP command is received, or a timeout occurs. This drops the state machine back to READY.

Commands the RSS to move the chest to a given pose. This command must be sent repeatedly, or rssdrive will timeout and stop motion.

This command is only valid in the state DRIVING. To enter DRIVING, send a ROBOT_SUPPORT_COMMAND with command DRIVE. Insertion stops when a STOP command is received, or a timeout occurs. This drops the state machine back to READY.

Keys:
  • position (double[3]) – The target chest position to move to.
  • orientation_quat (double[4]) – The target chest orientation to move to, as a quaternion.

Either one of these keys may be omitted to only perform translation or rotation.

ROBOT_SUPPORT_MOVE_CHEST_POSE_RESET¶

Commands the RSS reset its position or orientation. This command must be sent repeatedly, or rssdrive will timeout and stop motion.

This command is only valid in the state DRIVING. To enter DRIVING, send a ROBOT_SUPPORT_COMMAND with command DRIVE. Insertion stops when a STOP command is received, or a timeout occurs. This drops the state machine back to READY.

Keys:
  • reset_position (bool) – If true, moves the target chest position towards RSS Base frame y-axis without changing plunge.
  • reset_orientation (bool) – If true, moves the target chest rotation towards the identity quaternion.

Both of these keys should be in this message

ROBOT_SUPPORT_EXTERNAL_ROTATION¶

Commands the RSS to move the chest with an external rotation, meaning a rotation about the trocar. This command must be sent repeatedly, or rssdrive will timeout and stop motion.

This command is only valid in the state DRIVING. To enter DRIVING, send a ROBOT_SUPPORT_COMMAND with command DRIVE. Insertion stops when a STOP command is received, or a timeout occurs. This drops the state machine back to READY.

Keys:external_rotation_velocity (double[3]) – The angular velocity vector (in the chest frame) with which to pivot about the trocar. A +X velocity will make the chest pitch, +Y would roll, and +Z would yaw. This induces a linear and rotational velocity of the chest pose.

The message must have the keys external_rotation_velocity

ROBOT_SUPPORT_COMMANDED_CONFIGURATION_NAN_COUNTER¶

Publishes an array of counts of NANs in configuration that were intended to be commanded to the MCB.

Keys:counts (int[7]) – NANs caught and filtered out of configurations before being sent to the MCB for each configuration value.
ROBOT_SUPPORT_AUTOTRAVEL_PARAMETERS¶

Publishes parameters of the frustum in Travel-Pivot mode that can be used to visualize the cone of Travel/Pivot constraints (i.e. where chest motion will be activated). Note that the frustum shrinks when one of the constraints as been violated to prevent rapid chattering of violated / non-violated constraints. This message is used to provide visualization of the cones for debugging in RD_UI.

These keys are legacy keys for previous iterations of travel mode, and are kept for making rd_ui backwards compatible. This corresponds to the no motion zone: :key double frustum_far: The distance from the chest center that the midpoint must exceed for the chest to begin translating towards the midpoint :key double frustum_near: The distance from the chest center that the midpoint must be below for the chest to begin outserting along the trocar direction. :key double frustum_angle: The maximum angle at which the chest will begin rotating or translating to recenter the midpoint in front of the chest. This angle is defined by the grasper midpoint, the center of the chest, and the forward (+Y) direction of the chest.

The following are added to support multiple cones for new versions of travel / pivot mode. :key object[] cones_parameters: An array of objects which each define an elliptic cone. Each object in this array has four keys:

  • edge_far The forward boundary of this elliptic cone.
  • edge_near The rearward boundary of this elliptic cone.
  • ellipse_horizontal_angle The horizontal angular width of this elliptic cone.
  • ellipse_vertical_angle The vertical angular width of this elliptic cone.

Each cone represents a different region. The behavior in travel / pivot mode is determined by which one of these regions the midpoint of the graspers is in. The first cone is the no motion zone. While the midpoint is in this zone, the target chest pose will not move. The second cone is the travel only ellipse. When the midpoint is in this zone and outside of the no motion zone, the chest will translate to recenter the midpoint in the no motion zone.

Publishers:rssdrive – Publishes this topic when going into Autotravel mode or whenever the frustum changes within Autotravel mode.
